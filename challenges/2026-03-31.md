# AWS Privilege Escalation via Misconfigured GitHub Actions OIDC Trust

**Date:** 2026-03-31
**Topic:** Cloud Security
**Difficulty:** Hard
**Estimated Time:** 1–1.5 hours

## Scenario

Your organization deploys to AWS using GitHub Actions with keyless authentication via an IAM OIDC identity provider. During a routine access review, you discover the trust policy on the `github-actions-deployer` IAM role uses a dangerously broad subject condition:

```json
"Condition": {
  "StringLike": {
    "token.actions.githubusercontent.com:sub": "repo:my-org/*:*"
  }
}
```

This means *any* repository under `my-org` — including newly created ones or forks — can assume this role. The `github-actions-deployer` role has `lambda:*` and `iam:PassRole` permissions scoped to a wildcard resource (`"Resource": "*"`), plus read access to several S3 buckets containing customer PII exports.

A threat actor who gains the ability to push a GitHub Actions workflow to any repo in `my-org` (e.g., via a compromised developer account, a malicious pull request merged to a public fork, or by creating a new repo in the org) can exploit this to escalate to full `AdministratorAccess` in the production AWS account. Your job is to trace the complete attack path, validate it in a sandbox, harden the controls, and build detection coverage.

The environment: AWS (us-east-1), GitHub org `my-org`, an existing Lambda execution role `arn:aws:iam::123456789012:role/lambda-exec-admin` that has `AdministratorAccess` attached (created by a developer "just for testing" six months ago and never cleaned up).

## Objectives

- [ ] Map the full privilege escalation chain: from OIDC token in a rogue GitHub Actions workflow → `AssumeRoleWithWebIdentity` on `github-actions-deployer` → `iam:PassRole` + `lambda:CreateFunction` + `lambda:InvokeFunction` → code execution as `lambda-exec-admin` → `AdministratorAccess`. Document each API call and the IAM action it relies on.
- [ ] Harden the OIDC trust policy to restrict role assumption to specific repositories and refs (e.g., only `repo:my-org/infra-deploy:ref:refs/heads/main`), and apply an IAM permission boundary on `github-actions-deployer` that prevents `iam:PassRole` to any role with `AdministratorAccess` or `iam:*` policies attached.
- [ ] Write two CloudTrail-based detection rules (as EventBridge patterns or Sigma rules) that would alert on: (a) `AssumeRoleWithWebIdentity` from an unexpected repository subject, and (b) `CreateFunction` immediately followed by `InvokeFunction` where the passed execution role has admin-level permissions.
- [ ] Identify what legitimate CI/CD permissions the `github-actions-deployer` role actually needs and produce a least-privilege policy that replaces the current wildcard `lambda:*` + `iam:PassRole` grant.

## Hints

1. The `iam:PassRole` privilege escalation via Lambda is one of the most documented AWS privesc paths — Rhino Security Labs' enumeration is a good reference. The key constraint is that `iam:PassRole` alone is not enough; you also need a service (`lambda:CreateFunction`) that accepts the passed role and an invocation method (`lambda:InvokeFunction` or an event trigger) to actually execute code under the elevated identity. Trace which three IAM actions are the minimum required.
2. For the OIDC trust policy fix, the `sub` claim from GitHub follows the pattern `repo:<org>/<repo>:ref:refs/heads/<branch>` for branch-triggered workflows and `repo:<org>/<repo>:environment:<env>` for environment-gated ones. Use `StringEquals` (not `StringLike`) and pin to the exact repository and branch. Also consider adding a condition on `token.actions.githubusercontent.com:aud` to ensure the audience matches your expected value.
3. For the CloudTrail Sigma rule on (b), correlate events using `requestParameters.functionName` as a join key across `CreateFunction` and `Invoke` events within a short time window. The execution role ARN appears in `requestParameters.role` on `CreateFunction` — you can query IAM (or maintain a local cache) to check whether that role has `AdministratorAccess` or any `iam:*` actions in its attached policies.

## Review Questions

1. The OIDC `sub` claim from GitHub encodes the repo, ref, and trigger type, but it is ultimately controlled by GitHub's token issuance logic. What trust assumptions does your AWS account make about GitHub as an IdP, and what would the impact be if GitHub's OIDC endpoint were compromised or if an attacker found a way to forge a `sub` claim for a protected repo?
2. Permission boundaries cap what a role *can* do but do not grant permissions on their own. In what scenarios would a permission boundary on `github-actions-deployer` fail to prevent the Lambda privesc — for example, if the boundary is applied after the role is already compromised mid-workflow, or if the attacker can call `iam:PutRolePermissionsBoundary` themselves?
3. How would your detection strategy differ if the attacker used an existing Lambda function (i.e., `lambda:UpdateFunctionCode` + `lambda:UpdateFunctionConfiguration`) instead of creating a new one — and which CloudTrail event would you pivot on to catch that variant?
